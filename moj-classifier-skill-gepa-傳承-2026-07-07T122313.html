把「檢察官書類分類」教會一顆地端 31B 小模型
—— GEPA 疊代 × Dynamic Workflow × pi.dev 技能:全程傳承(2026-07-07)

這一頁記錄一件事怎麼從零做到驗收:讓一顆跑在地端 GPU 的 31B 小模型(經由 pi -p 驅動),讀懂檢察官起訴書/不起訴處分書,自動分類到五種案件類型並填好統計欄位, 最後吐出 5 張 CSV 與 1 份 Excel。訓練分從 88.68 進化到 98.15,從未見過的保留題考 91.13, 100 個檔案冷跑只要 2.7 分鐘。

001|人類原始提示全文(順稿整理版)

原始提示為口述逐字稿,錯字、贅詞與標點已依「不改原意」原則整理; 結構分段為後加。共七段提示,依時間順序編號。

【提示 001|開場需求(口述整理)】 我們在 data 底下有 input 跟 output。input 裡面每一個資料夾都是一批案件,可能是 PDF、可能是 DOCX,是每一個人的起訴、不起訴跟偵查結果的書類;裡面會有個資、犯罪行為等等。 一開始系統是沒有「人口販運案件、未遂犯、幫助犯、重大刑事案件、國土保育案件」這些資料夾的,連資料夾名稱都不知道——是人類專家判斷之後,才幫它分好這些資料夾。 分好資料夾之後,還要對應 output 資料夾裡的 Excel 表:它有五個頁簽,對應剛剛人工建立的五個分類。每一筆個案都要有上面的欄位——例如它是臺北地檢還是桃園地檢、年份是 114 年、字別是偵字還是偵緝字(這些都有機關自己的代號)、被告姓名(去識別化)——最重要的是要判它屬於起訴、不起訴還是緩起訴。 每個頁簽下面要分類的項目又不一樣:人口販運是「多選題」,有性剝削、勞力剝削、器官摘取、人流處置;國土保育通常是「單選題」,可能是野生動物保育法或森林法;幫助犯也有一些類別。 這些 know-how 放在「客戶提供的原始資料-統計書類」裡,有兩個檔案:一個 TXT 是領域專家口述的前情提要(ru 口述前情提要目標.txt);另一個是人類專家訪談事務官整理出來的 HTML(17-統計處書類裁判書標籤判讀),裡面寫了 AI 要怎麼判讀起訴書/不起訴書、怎麼判未遂、怎麼判幫助犯(例如刑法第 30 條)。重大刑案會有指定法條清單;國土保育也有指定條號,例如森林法第 51、53、54 條——52 條就不是。每一個欄位都是專家指定的法條專業領域,這些都很重要。 現在請你上網查 pi.dev 怎麼製作技能。你的電腦裡已經有 pi,用「pi 空格 -p」可以驅動地端 31B 模型。查清楚之後,在 .pi/ 底下可以做 skills。 針對輸入輸出的需求,你要整理出乾淨的資料夾:訓練資料要切割成「訓練集」跟「保留集」,放不同資料夾。Fable(你)有能力做出標準正確答案;但技能訓練的目標對象是 pi -p 的 31B 小模型。我希望你用 Fable 把資料夾轉成 AI 友善的純文字——PDF 轉純文字、Excel 也轉 AI 友善格式。一旦轉好,你就可以自由切割保留集和訓練集來訓練技能。 訓練技能時,請上網查一個叫 GEPA 的 GitHub。GEPA 可以多輪疊代:評審請用 Fable subagent、乾淨上下文來打分;然後照 GEPA 的精神,每一輪針對 pi -p 這個「選手」疊代。GEPA 就像教練,不斷教選手進步。技能越進步,它對輸入輸出的考題就越厲害。 所以你要清楚告訴我:考題有幾題、訓練集訓了幾題、訓練完保留題過了幾題、幾分、情況怎麼樣。評審、裁判用 Fable subagent 做,選手是 pi -p。每一輪疊代都要清清楚楚講:幾分、做了什麼事;疊代過程的技能或腳本都要留下來。技能如果純文字 SKILL.md 就能跑,就這樣跑;要加腳本的話,pi.dev 可以加技能腳本,請用 bun 寫 TypeScript 放進去。 技能疊代完成之後,它就有能力把 AI 友善的純文字輸入,做成 AI 友善的純文字輸出——輸出就是剛剛講的試算表,每個欄位都要有,等於是把一個案件做標籤分類,輸出成不同頁簽的試算表。最後你再用腳本工具把它變成真正的 Excel 格式。工具一律優先用 bun + TypeScript 製作。每一輪都報分數和原因。 【提示 002|目標鎖定】 請用 GEPA 進化到大於 90 分才停止。 【提示 003|進度關切】 請告訴我現在進度,你是卡住了嗎?還是還在做? 【提示 004|Dynamic Workflow 重構指示(口述整理)】 有幾個問題要講。你的技能出來之後,很有可能要搭配腳本做多步驟產出——當然包含 OCR、Word、PDF 轉 AI 友善純文字,這些你都要做,而且要包含在技能裡面;不能搞半天這些不在技能裡,那就穿幫了。所以技能一定要有「AI 友善轉換」的能力,轉換完最後還要變成 Excel。 問題來了:31B 在做這些的時候,請你上網查 Claude Code 的 Dynamic Workflow。它的原理是用 JS 腳本控制整個工作流——技能在腳本下面被控制。多步驟、多腳本,就有機會用 JS 控制工作流,不會讓 Agentic AI 亂掉、有步驟沒做到、做得不夠好、或沒有平行處理。這些都可以用 Dynamic Workflow 的原理解決。 我們希望你把這個概念用在本案:31B 的技能啟動之後,有一個 bun TypeScript 輔助腳本實現 Dynamic Workflow。腳本裡做多步驟拆解:從原始資料開始,要怎麼呼叫大模型——可以用 pi 空格 -p 的方式呼叫,因為它有 Agentic AI 的能力——而且最好平行做,不然會很慢。透過這些方式,就有機會把大量資料處理好。 另外一個困惑:如果資料不是只有幾個檔案,而是十幾個、100 個、300 個檔案,你的壓力測試過得了過不了?要跑多久?所以做完之後,幫我測:多快(速度數字)、多省資源、轉換成最終檔案多精準、品質多好——給我一份測量報告。每一輪疊代時,要根據這些 baseline 跟測量報告的差異,才能疊代下一代。 我猜最後實際使用也不會一個檔案一個檔案分類,那樣沒意義;應該是批次做,一次 100 或 300 個檔案。例如一個地檢署多個檔案全部放進某個資料夾,技能一啟動,就跑 Dynamic Workflow 裡寫好的腳本完成這件事。總之怕你工作流設計錯誤——上網查清楚之後,照 Claude Code Dynamic Workflow 的精神,好好壓力測試、重構成能夠很穩健地用 31B、用 pi -p 完成的東西。 【提示 005|記分板在哪】 請問你每一輪的分數打在哪裡?我沒看到選手的分數。 【提示 006|輸出格式確認】 很厲害的東西。對方本來就有一個標籤管理系統,最後要把試算表 Excel 匯回去。所以你的產出可不可以是多張 CSV?照理來講應該有五個,對應五個頁簽:重大刑事案件、人口販運案件、國土保育案件、幫助犯、未遂犯。同時也要產出真正的 Excel。因為你用 bun + TypeScript,不管 CSV 或 Excel,產出這些檔案之後就可以匯回原來的系統。介接傳輸已經透過 Files API 解決,系統之間可以做檔案交換——所以你只要負責產生檔案就好,介接傳輸不用你負責。 【提示 007|傳承指令】 把這一次做分類技能的整個過程 prompt 做一個傳承。prompt 的部分我要原文,但因為原文有錯字、贅詞、錯標點,所以你要順稿整理、結構分段。

Round 0|初版技能:88.68 分

做法:把專家 know-how(法條清單、判讀原則)加上從訓練集歸納的代碼對照表,寫成一份純文字判讀提示,讓 31B 讀完書類直接輸出欄位 JSON。

結果:訓練集 60 題平均 88.68。人口販運只有 75.69;姓名欄錯了 41 件。

演進理由:沒過 90 門檻,失分集中在三處:多案號選錯、人口販運判碼混亂、姓名遮罩錯字數。

Round 1|教練第一次反思:92.39 分(+3.71)

做法:把 46 個錯題整理成「失敗摘要」(期望 vs 實際+書類節錄),交給一位乾淨上下文的 Fable 教練 subagent 診斷。教練歸納出「多案號取最新年度那一組、組內取第一個」的規律,並把人口販運改成法條驅動判碼。

結果:92.39,人口販運 75.69 → 90.28。首次跨過 90。

演進理由:姓名欄還是錯 40 件——教練寫了更詳細的遮罩規則、更多正反例,但完全沒用。這暴露了一個本質問題(見下一輪)。

Round 2|Dynamic Workflow 關鍵一刀:97.29 分(+4.90,最大單輪躍升)

做法:診斷出 31B 的天生短板:它數不對中文字有幾個字。「王大明」該遮成「王○○」, 它常輸出「王○」——不管規則寫得多細都一樣,因為這是斷詞層的限制,不是理解問題。解法不是繼續教, 而是「不讓它做」:模型只負責照抄全名,遮罩改由腳本機械執行。

結果:97.29,姓名錯 40 件 → 3 件,未遂犯類滿分 100。

演進理由:這就是使用者指示的 Dynamic Workflow 精神:凡是有確定規則的事(遮罩、驗證、補零、重試、平行),都放進腳本;模型只留語意判斷。

Round 3 + 變異數檢測:97.99 / 98.13

做法:教練第二次反思(is_expn 改「只看論罪法條,事實關鍵詞不得勾碼」鐵律、選被告先排除告訴人)。 但發現兩案從滿分變錯——是真回退還是模型隨機性?同一份提示原封不動重考一次。

結果:97.99 與 98.13,整體變異只有 ±0.14;但那兩案兩輪都錯 → 確認是真回退,不是雜訊。

演進理由:疊代要有統計紀律:先分清「雜訊」與「訊號」,才知道下一刀切哪裡。

Round 4|手術式合併:98.15 分

做法:不整份重寫,只把回退案缺的兩條規則補回去:①兒少條例條號集合強制檢查 {32,33,34,37};②毒品案必須把附表數量逐筆加總,合計 ≥1000 公克才算重大刑案(部分未遂仍算重大)。

結果:98.15,毒品加總案修復。

演進理由:GEPA 的「系統感知合併」:保住各候選版本各自答對的部分,只動壞掉的段落。

Round 5|名稱陷阱規則+收斂:98.15 分(定版)

做法:最後一個頑固錯題揭露了 31B 的第二個盲點:法規名稱「兒童及少年性剝削防制條例」 含「性剝削」三個字,模型看到名字就直接判人口販運——加上「法規名≠分類,必查條號」警告。

結果:98.15 持平;逐案 Pareto 比較與前版各勝 1 案、58 平。已觸及雜訊天花板 (約 98.5:剩 7 件是 gold 登錄噪音或姓氏根本不在文件裡)。宣告收斂。

去識別化(整支 mask.ts,這 20 行救回 37 個錯題):

// 姓名去識別化(確定性腳本邏輯,不交給小模型數字數)
export function maskName(name: string): string {
  const s = String(name ?? "").trim().replace(/\s+/g, " ");
  if (!s) return s;
  if (s.includes("○")) return s.replace(/\s/g, "");     // 已匿名:照抄
  if (/^[A-Za-zÀ-ÿ'. -]+$/.test(s)) {
    return s.split(" ")[0] + "○○";                        // 拉丁姓名:首字+○○
  }
  const chars = [...s.replace(/[\s·]/g, "")];
  const L = chars.length;
  if (L <= 1) return s;
  if (L === 2) return chars[0] + "○";                     // 2 字名
  if (L === 3) return chars[0] + "○○";                    // 3 字名
  return chars[0] + chars[1] + "○".repeat(L - 2);         // 4 字雙姓
}

③ 最終判讀提示(classify.md 結構;範例姓名均為示意化名)

輸出格式:單一 JSON,含 category 與 iv_orgn/iv_year/iv_iden/iv_srno/iv_name/iv_dtst 等欄位(代碼+文字成對)。

步驟 1 選案號:多案號並列 → 只留「年度最新」那組 → 組內取第一個;案號左補零到 6 碼(3849→003849,絕不是 038490)。

步驟 2 基本欄位:機關代碼表(臺北 2213/桃園 2214/新北 2216/臺中 2221);書類種類 → 處分代碼(起訴書=10、追加起訴=13、聲請簡易判決=20、不起訴=30、緩起訴=36)。

步驟 3 被告姓名:照抄全名不要自己加○(遮罩由後端程式處理);跳過純代號被告;用「原名」時取現用名;已匿名書類照抄「甲○○」。

步驟 4 分類(依序檢查,命中即停):人口販運 → 國土保育 → 重大刑事 → 幫助犯 → 未遂犯。 三個鐵律:①「未遂/幫助」不會讓案件掉出前面的類別;②兒少條例必查條號 ∈ {32,33,34,37},法規名稱含「性剝削」不代表人口販運;③毒品案必把附表數量逐筆加總,≥1000 公克才算重大刑案。

步驟 5 人口販運五碼:法條驅動——只看選定被告論罪段引用的法條查表勾碼;犯罪事實裡的「招募、載送、使人出國」等關鍵詞一律不得作為勾碼依據(刑法§297/§302、組織犯罪條例都不觸發)。

步驟 6-7 案由與法條代碼:未遂犯跟著「未遂那條罪」;幫助犯跟著「被幫助的本罪」(絕非刑法 30 條);同時幫助詐欺+洗錢從一重取洗錢;法條代碼 9 碼構造公式+固定對照表。

突破一:把「模型不擅長的事」搬出模型(+4.90 分,全程最大單刀)。 前兩版卡關,是因為一直想「教會」31B 數中文字數——但那是斷詞層的天生限制,教不會。 突破點是換問題:不問「怎麼教會它遮名字」,改問「這件事誰做最穩」。答案是 20 行確定性程式。 (真操作:提示改成照抄全名,mask.ts 在比對與產出前統一遮罩。)

突破二:失敗軌跡餵給乾淨上下文的教練,而不是總分。 GEPA 之所以有效,是教練看得到「錯在哪一欄、期望什麼、拿到什麼、原文長怎樣」, 才能歸納出「多案號取最新年度組第一個」「法規名稱含性剝削≠人口販運」這種人看得懂、模型守得住的規則。 只給總分的話,教練只能瞎猜。(真操作:failures.ts 自動產失敗摘要;教練 subagent 讀原始書類交叉驗證假說後才動筆。)

突破三:疊代有統計紀律,才敢收斂。 第 3 輪出現「改善但有回退」,先重考同一份提示分辨雜訊與訊號(變異 ±0.14,回退是真的), 再手術式只補缺的規則、不整份重寫;最後用逐案 Pareto 比較確認觸頂(剩 7 件是標註噪音與文本裡根本沒有的答案), 才宣告收斂並開封保留集。保留集 91.13 證明沒有過擬合——差距全來自「沒背過的代碼」, 而那正是生產環境會由官方對照表補上的部分。

給接手者的三個坑: ① gold Excel 的 iv_idno 在書類文本中不存在,別拿來計分; ② 3 件書類文本已匿名(甲○○)但 gold 留了真實姓氏——模型不可能答對,建議修 gold; ③ 雲端 OCR 閘道會 502,本機 macOS Vision 對繁中印刷書類又快又準,優先用它。
